# RELIABILITY — 障害耐性設計・ログ・リトライ方針

目的
- システムの可用性・観測性を高め、障害時の復旧手順と挙動を明確にする。

ロギングと観測
- 構造化ログ（JSON 推奨）を採用し、ログレベル（debug/info/warn/error）を明確にする。
- 相関 ID を導入し、1 バックテスト実行に紐づくログを追跡できるようにする。
- 出力ディレクトリ命名規則: python の出力は reproducible なパスを使う（例: python/output/backtest/backtest_YYYY-MM-DD_to_YYYY-MM-DD_TIMESTAMP/）。

リトライ方針
- 一時的な障害に対しては指数バックオフ+ジッターを用いた再試行を行う。最大試行回数とタイムアウトは明示する。
- 冪等性を保証すること（同じリクエストを複数回送っても副作用が重複しない設計）。

フェイルオーバーと回復
- 長時間処理は進捗と中間成果を定期的に永続化し、途中からの再開を可能にする。
- クリティカルなジョブは監視アラートを設け、閾値（失敗率・遅延）を超えたら通知する。

ヘルスチェックと自己診断
- API と長時間ジョブはヘルスチェックエンドポイントを提供する。
- スモークテストを定期実行し、主要ワークフローが正常かを監視する。
