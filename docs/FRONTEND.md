# FRONTEND — フロントエンド実装規約・状態管理ルール

目的
- フロントエンド実装で従うべき技術スタック、状態管理、テスト、アクセシビリティ、パフォーマンスの指針を定める。

技術スタック
- React + TypeScript を採用。型は厳格（strict）を推奨する。
- スタイリングは Tailwind / design tokens を基本とする（既存の tailwind.config.js を参照）。
- ビルド／テストコマンド例：
  - ビルド: npm --prefix frontend run build
  - 単体テスト: npm --prefix frontend run test

状態管理
- ローカル状態: 一時的な UI 状態は useState/useReducer を使用。
- グローバル状態: アプリ全体で共有する必要のある状態は中央ストア（Redux, Zustand 等）で管理。データは正規化し、セレクタ経由で取得する。
- サーバー由来のデータはキャッシュライブラリ（React Query 等）を使って整合性と再試行を管理するが、決定的なビジネスロジックはバックエンドで実行する。

API 契約とデータ
- API の契約に厳密に従う。特にチャートは backend が base64 画像で返す場合がある（data:image/png;base64,...）ため、契約変更時は backend の型定義と frontend のテストを同時更新する。
- フロントは数値のフォーマット・ローカライズのみ行い、集計やシミュレーション等のロジックは行わない。

テスト
- ユニット／コンポーネント／統合テストを整備。外部 API は必ずモックし、ネットワークに依存しない。
- アクセシビリティ検査（axe 等）を CI に組み込むことを推奨。

パフォーマンスと最適化
- コード分割、遅延読み込みを行い初期バンドルを小さく保つ。
- 重い描画は virtualization を使う。

アクセシビリティ
- キーボード操作、スクリーンリーダー対応、適切な ARIA の使用を必須とする。

運用とデプロイ
- 環境変数は REACT_APP_API_URL を使って API エンドポイントを切り替える。
- 重要な UI 変更はスクリーンショットと簡易ガイドを PR に添付すること。
