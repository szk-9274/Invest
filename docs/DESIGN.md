# DESIGN — UI/UX 設計原則と視覚的一貫性

目的
- ユーザーにとって直感的で一貫性のある体験を提供するための設計原則と実装上の注意点を定義する。

基本原則
- 明快さ（Clarity）：情報は意味のある優先順位で提示する。重要な数値やアラートは目立たせ、計算の由来（入力パラメータ・期間など）を明示する。
- 一貫性（Consistency）：デザイントークン（色、間隔、タイポグラフィ、アイコン）をプロジェクト全体で再利用する。UIコンポーネントは同一のパターンに従う。
- アクセシビリティ（Accessibility）：WCAG AA を目安にコントラスト、フォーカス順、キーボード操作、代替テキストを確保する。
- レスポンシブと可変レイアウト：モバイルファーストで設計し、主要ブレークポイントごとにレイアウトを検証する。
- 最小限の運動（Motion）：アニメーションは状態変化の可視化に限定し、過度な装飾を避ける。

コンポーネント設計
- 単一責任：コンポーネントは表示に専念し、ビジネスロジックはコンテナ層またはバックエンドに置く。
- プレゼンテーションとデータの分離：データ整形は可能な限りバックエンドで行い、フロントは受け取った契約を描画するのみとする。
- チャート表示ルール：チャートは軸ラベル、凡例、ツールチップ、期間の明示を必須とする。バックエンドは画像を data:image/png;base64,... の形で返すことがあるため、フロントはこの形式を前提に表示する（契約変更時は型とテストを同時に更新）。

デザイン資産と運用
- デザイントークンは frontend 側（tailwind.config.js 等）に集約し、色・間隔・タイポは変数として再利用する。
- 変更時はスクリーンショットを PR に添付し、視覚回帰テスト（ある場合）を実行する。

検証とレビュー
- 大きな UI 変更はデザインレビューを行い、UX の観点から影響範囲（表、チャート、レイアウト）を確認する。
- 重要な可視化の変更は docs/STRATEGY.md や関連ドキュメントと整合性を取る。

例（輝度・配色の最小ルール）
- Primary / Accent / Background / Surface / Error を明示し、アクセシビリティを満たす色組を用いる。

（補足）
- フロントは計算ロジックを持たず、バックエンドの出力（CSV/JSON/チャート）を表示することを原則とする。
