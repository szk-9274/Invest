# QUALITY_SCORE — コード品質指標・テスト基準

目的
- コード品質を定量化し、リリース前に満たすべき最低基準と推奨基準を定義する。

必須基準
- テスト: 新規機能は必ずテストを先に書く（TDD）。外部 API は全てモックすること。
- 実行時間: 単体テストは可能な限り短く保つ（目安: 全体で数十秒、個々のテストは 5 秒以内を推奨）。
- カバレッジ: 重要ロジック（バックテスト・スクリーニング）は高いカバレッジを維持（目標 80% 以上、クリティカル箇所は 90%）。

契約テスト（出力の契約）
- CSV/DataFrame の必須カラム例（変更時は契約テストを追加）:
  - 共通: ticker, exchange, sector, stage
  - バックテスト出力（例）: trades.csv には timestamp, ticker, action, size, price, pnl が含まれること
- 契約テストは CI で必ず実行する。

静的解析と型検査
- TypeScript は strict モード、Python は mypy 等で型チェックを行う。
- linters（ESLint, flake8 等）を CI に組み込み、低レベルのスタイル違反は CI でブロックする。

CI ゲート
- マージ前に以下を満たすこと:
  - 全テスト通過
  - 型チェック通過
  - 重要な契約テスト通過

運用上の注意
- テストはネットワークに依存しないこと。外部依存は全てモックする。
